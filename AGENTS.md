# Agent Guide (btcwallet-sql-tx-adr)

This repository is Go-first with SQL-backed stores generated by `sqlc`.
Prefer using the `Makefile` targets so you match CI and local tooling.

Authoritative rules for this repo live in:

- `docs/developer/ENGINEERING_GUIDE.md`
- `docs/developer/code_formatting_rules.md`
- `docs/developer/unit_testing_guidelines.md`

No Cursor rules (`.cursor/rules/`, `.cursorrules`) and no Copilot rules
(`.github/copilot-instructions.md`) are present in this repo at the time of
writing. If those are added later, update this file to reflect them.

## Commands

- Build: `make build`; compile gate: `go install -v ./...`
- Format (Docker): `make fmt` / `make fmt-check`
- Lint (Docker): `make lint` (auto-fix) / `make lint-check` (`workers=4` to limit)
- SQL (Docker + sqlfluff): `make sql-parse`, `make sql-format[-check]`,
  `make sql-lint[-check]`
- sqlc (Docker): `make sqlc` / `make sqlc-check`

## Tests

- Unit tests (all): `make unit`
- Unit tests (single pkg): `make unit pkg=wallet/internal/db`
- Unit tests (single case regex):
  `make unit pkg=wallet/internal/db case=TestPostgresStore_CreateWallet`
- Unit compile-only: `make unit pkg=wallet/internal/db case=_NONE_`
- Common flags: `verbose=1`, `nocache=1`, `timeout=5m`, `log="info stdlog"`
- DB itests (Docker): `make itest-db db=sqlite|postgres`
  (coverage: `cover=1`, race: `make itest-db-race db=...`)
- Full wallet itests: `make itest chain=<btcd|neutrino|bitcoind> db=<kvdb|...>`
  (filter: `icase=<regex>`)

## SQL Layout (Migrations + Queries + Generated)

- Migrations: `wallet/internal/db/migrations/{postgres,sqlite}/NNNNNN_*.sql`
  (`*.up.sql` non-idempotent; `*.down.sql` idempotent)
- Queries: `wallet/internal/db/queries/{postgres,sqlite}/*.sql`
- Generated (do not edit): `wallet/internal/db/sqlc/{postgres,sqlite}/`
- Workflow: edit SQL -> `make sql-format sql-lint` -> `make sqlc`

## Go Code Style

Formatting and imports:

- Use `make fmt` (it runs `gosimports` and `gofmt`).
- Tabs for indentation (`.editorconfig`).
- Keep lines short (target ~80 chars; tab width is 8 spaces).
- Imports are grouped: stdlib, blank line, third-party, blank line, repo.
- Side-effect imports are allowed only when required (e.g. SQL drivers); keep
  the `_ "pkg"` import with a short comment explaining why.

Spacing and wrapping (from `docs/developer/code_formatting_rules.md`):

- Break functions into logical stanzas separated by blank lines.
- Wrap long function calls by putting `(` at the end of the call name, then one
  arg per line, and `)` on its own line.
- For `fmt.Errorf`/`log.*f` formatting calls, prefer compact wrapping to reduce
  visual noise.
- If a function signature spans multiple lines, start the function body with an
  empty line.

API and naming:

- `context.Context` is the first parameter; pass `ctx` through and honor
  cancellation.
- Use repo naming conventions (examples: `Txid`, `UTXO`, `PkScript`).
- For SQL-facing IDs, follow existing contracts (often `uint32` in domain
  types, converted to/from `int64` at the DB boundary).
- Avoid silent narrowing casts; use the safe-casting helpers in
  `wallet/internal/db/safecasting.go` when converting from SQL ints.

Errors:

- Sentinel errors: `var ErrX = errors.New("...")` (lowercase, no punctuation).
- Wrap errors with context: `fmt.Errorf("do thing: %w", err)`.
- Use `errors.Is`/`errors.As` for checks; map `sql.ErrNoRows` to domain errors.
- Prefer returning actionable context in the error string (what failed + why).

Transactions and DB access:

- Use `(*PostgresStore).ExecuteTx` / `(*SqliteStore).ExecuteTx` for atomic
  operations.
- Keep transaction closures small; do not perform network I/O inside a DB
  transaction.
- Prefer adding SQL to `queries/*.sql` and using sqlc over ad-hoc SQL strings
  in Go (itest helpers are the main exception).

Documentation:

- Exported identifiers must have GoDoc comments that start with the name.
- Prefer full-sentence comments and focus on invariants, ordering, and why.
- In tests, use `t.Helper()` in helpers and prefer `t.Context()`.

Testing style (from `docs/developer/unit_testing_guidelines.md`):

- Use AAA structure (Arrange/Act/Assert) separated by blank lines.
- Prefer `require` over `assert` for setup and critical checks.
- Use table-driven tests only when the Arrange structure is identical for all
  cases; avoid conditional setup inside the `for` loop.
- Keep table-driven test structs data-only (no setup closures).
- Prefer standalone tests for complex orchestration.

## SQL Style

Follow `.sqlfluff`:

- Keywords UPPERCASE; identifiers lowercase; types UPPERCASE.
- Indent with 4 spaces.
- Prefer `!=` over `<>`.

Keep Postgres/SQLite behavior aligned unless there is an engine-specific reason
(document any divergence in comments and in the relevant ADR/docs).

## Generated Code

- Do not hand-edit `wallet/internal/db/sqlc/**`.
- Do not hand-edit generated RPC output under `rpc/walletrpc/**`.
- Regenerate via `make sqlc` / `make rpc` and commit the generated diffs.

## Engineering Notes

Key themes from `docs/developer/ENGINEERING_GUIDE.md` (non-exhaustive):

- Security-first, fail-safe defaults.
- Context propagation everywhere; avoid goroutine leaks.
- Actor-model style concurrency where possible.
- Wrap errors for context; use sentinel errors for domain conditions.
- Comments explain *why*, not *what*.
